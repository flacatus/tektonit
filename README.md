# tektonit

An autonomous LLM-powered agent that watches a GitHub repository containing Tekton CI/CD resources, generates BATS unit tests for every embedded script, and opens pull requests with passing tests.

It runs as a container in Kubernetes, continuously monitoring your Tekton catalog for resources without tests — then generates, validates, fixes, and proposes them via PRs.

## How it works

```
┌──────────────────────────────────────────────────────────┐
│                    tektonit (container)                   │
│                                                          │
│  1. Clone/pull your Tekton catalog from GitHub           │
│  2. Scan for Tasks, StepActions with untested scripts    │
│  3. Prioritize by complexity (most complex first)        │
│  4. For each untested resource:                          │
│     a. Generate BATS tests using LLM                     │
│     b. Evaluate with skeptical reviewer (multi-agent)    │
│     c. Run tests, diagnose failures                      │
│     d. Fix progressively (up to 10 attempts)             │
│     e. Check for flakiness (3 runs)                      │
│     f. Learn from outcome (episodic memory)              │
│  5. Open PR with passing tests                           │
│  6. Sleep, repeat                                        │
└──────────────────────────────────────────────────────────┘
```

## Quick start (Kubernetes)

### 1. Configure secrets

```bash
kubectl apply -f deploy/namespace.yaml
```

Edit `deploy/secret.yaml` with your actual keys:

```yaml
stringData:
  GITHUB_TOKEN: "ghp_your_token_with_repo_and_pr_permissions"
  GEMINI_API_KEY: "your_gemini_api_key"
```

```bash
kubectl apply -f deploy/secret.yaml
```

### 2. Configure the target repository

Edit `deploy/deployment.yaml` — set your catalog repo and preferences:

```yaml
env:
  - name: GITHUB_REPO
    value: "your-org/your-tekton-catalog"
  - name: REPO_BRANCH
    value: "main"
  - name: POLL_INTERVAL_SECONDS
    value: "3600"           # Check every hour
  - name: BATCH_SIZE
    value: "10"             # Resources per cycle
  - name: MAX_FIX_ATTEMPTS
    value: "3"              # Fix attempts per resource
```

### 3. Deploy

```bash
kubectl apply -f deploy/deployment.yaml
```

### 4. Monitor

```bash
# Watch logs
kubectl logs -n tektonit -l app.kubernetes.io/name=tektonit -f

# Check health
kubectl port-forward -n tektonit deploy/tektonit 8080:8080
curl http://localhost:8080/healthz
curl http://localhost:8080/metrics
```

The agent will start scanning your catalog and opening PRs with generated tests.

```bash
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"

# Scan a catalog (no LLM needed)
tektonit scan /path/to/tekton-catalog

# Generate tests for all resources
export GEMINI_API_KEY=your-key
tektonit generate /path/to/tekton-catalog

# Generate for a single resource
tektonit generate-single path/to/my-task.yaml

# Run generated tests
bats path/to/sanity-check/*.bats
```

## What it generates

For each Tekton resource with embedded bash scripts, tektonit creates a BATS test file in `sanity-check/` next to the YAML:

```
tasks/my-task/0.1/
  my-task.yaml
  sanity-check/
    my_task_unit-tests.bats    <-- generated by tektonit
```

Each test file:
- Embeds the script verbatim via heredoc
- Stubs all Tekton variables (`$(params.X)`, `$(results.X.path)`)
- Mocks every external command (`kubectl`, `curl`, `jq`, `oras`, etc.)
- Tests all code paths: success, failure, edge cases
- Verifies exit codes, output strings, and result file contents
- Works on both macOS and Linux

### Example generated test

```bash
#!/usr/bin/env bats

setup() {
    export MOCK_BIN="$BATS_TEST_TMPDIR/mock_bin"
    mkdir -p "$MOCK_BIN"
    export PATH="$MOCK_BIN:$PATH"

    # Mock kubectl
    cat > "$MOCK_BIN/kubectl" << 'MOCK_EOF'
#!/usr/bin/env bash
if [[ "$*" == *"get pods"* ]]; then
    echo '{"items": [{"metadata": {"name": "pod-1"}}]}'
fi
MOCK_EOF
    chmod +x "$MOCK_BIN/kubectl"

    # Extract and prepare script
    cat > "$BATS_TEST_TMPDIR/script.sh" << 'SCRIPT_EOF'
#!/usr/bin/env bash
set -euo pipefail
PODS=$(kubectl get pods -n "$(params.namespace)" -o json)
COUNT=$(echo "$PODS" | jq '.items | length')
echo "[INFO]: Found $COUNT pods"
echo -n "$COUNT" > "$(results.pod-count.path)"
SCRIPT_EOF
    sed -i'' -e 's|$(params.namespace)|test-ns|g' "$BATS_TEST_TMPDIR/script.sh"
    sed -i'' -e "s|\\$(results.pod-count.path)|$BATS_TEST_TMPDIR/results/pod-count|g" "$BATS_TEST_TMPDIR/script.sh"
    mkdir -p "$BATS_TEST_TMPDIR/results"
    chmod +x "$BATS_TEST_TMPDIR/script.sh"
}

teardown() { rm -rf "$MOCK_BIN"; }

@test "counts pods correctly" {
    run "$BATS_TEST_TMPDIR/script.sh"
    [ "$status" -eq 0 ]
    [[ "$output" == *"[INFO]: Found 1 pods"* ]]
    [ "$(cat "$BATS_TEST_TMPDIR/results/pod-count")" = "1" ]
}

@test "fails when kubectl errors" {
    cat > "$MOCK_BIN/kubectl" << 'EOF'
#!/usr/bin/env bash
echo "error: connection refused" >&2; exit 1
EOF
    chmod +x "$MOCK_BIN/kubectl"
    run "$BATS_TEST_TMPDIR/script.sh"
    [ "$status" -ne 0 ]
}
```

## Autonomous agent capabilities

The agent has 8 intelligence capabilities that make it think like a senior test engineer:

| # | Capability | Description |
|---|---|---|
| 1 | **Episodic Memory** | Learns from past failures, stores patterns in SQLite, avoids repeating mistakes |
| 2 | **Failure Diagnosis** | Classifies failures (mock_mismatch, assertion_drift, timeout, etc.) before fixing |
| 3 | **Multi-Agent Separation** | Generator creates, skeptical evaluator critiques (avoids confirmation bias) |
| 4 | **Flaky Detection** | Runs tests 3x after passing to catch non-determinism |
| 5 | **Coverage Analysis** | Counts branches vs tests, requests more tests if coverage is low |
| 6 | **Risk-Based Prioritization** | Scores resources by complexity, processes highest-risk first |
| 7 | **PR Feedback Learning** | Harvests review comments from closed PRs, applies lessons |
| 8 | **Progressive Fix Strategy** | Targeted fix -> rewrite mocks -> full regeneration -> submit as-is |

### Progressive fix strategy

When tests fail, the agent doesn't blindly retry. It diagnoses first, then escalates:

```
Attempts 1-3:  TARGETED FIX     — Fix the specific diagnosed error
Attempts 4-6:  REWRITE MOCKS    — Keep tests, rebuild all mocking from scratch
Attempts 7-9:  FULL REGENERATION — Start over with failure context
Attempt 10:    SUBMIT AS-IS     — Mark CODE_ISSUE if script bug suspected
```

## Environment variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `GEMINI_API_KEY` | Yes* | - | Gemini API key |
| `GITHUB_TOKEN` | Yes | - | GitHub token with repo + PR permissions |
| `GITHUB_REPO` | Yes | `flacatus/tekton-integration-catalog` | Target repository (owner/repo) |
| `REPO_BRANCH` | No | `main` | Branch to monitor |
| `POLL_INTERVAL_SECONDS` | No | `3600` | Seconds between scan cycles |
| `BATCH_SIZE` | No | `10` | Max resources per cycle |
| `MAX_FIX_ATTEMPTS` | No | `3` | Max fix attempts per resource |
| `LLM_PROVIDER` | No | `gemini` | Provider: gemini, claude, openai |
| `STATE_DB_PATH` | No | `/var/lib/tektonit/state.db` | SQLite database path |
| `HEALTH_PORT` | No | `8080` | Health/metrics endpoint port |

*Or `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` depending on provider

## Observability

- **Prometheus metrics** at `/metrics` — tests generated, fix attempts, PRs created, cycle duration
- **Health checks** at `/healthz` (liveness) and `/readyz` (readiness)
- **Structured JSON logging** for log aggregation

## Architecture

```
tektonit/
  monitor.py           Production daemon — clone, scan, generate, PR, repeat
  test_generator.py    Autonomous pipeline — generate, evaluate, run, fix, learn
  parser.py            Parse Tekton YAML into structured dataclasses
  llm.py               LLM provider abstraction (Gemini, Claude, OpenAI)
  prompts.py           System prompts and prompt builders
  state.py             SQLite persistence (resources, memory, feedback)
  github_client.py     GitHub integration (clone, branch, commit, PR)
  resilience.py        Retry with backoff, circuit breaker, rate limiter
  observability.py     Prometheus metrics, JSON logging, health endpoints
  cli.py               Click CLI for local development
```
